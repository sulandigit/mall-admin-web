# Vuex Store结构优化迁移策略

## 当前架构分析

### 现有Store结构
```
src/store/
├── index.js          # Store根配置文件
├── getters.js        # 全局getters
└── modules/
    ├── app.js         # 应用状态模块
    ├── user.js        # 用户认证模块
    └── permission.js  # 权限管理模块
```

### 问题识别

1. **模块划分不够细化**
   - 缺乏按业务模块（PMS/OMS/SMS/UMS）划分的状态管理
   - 通用功能状态分散在各模块中

2. **状态耦合度较高**
   - getters集中在根级别，未按模块组织
   - 模块间缺乏明确的边界

3. **缺乏命名空间**
   - 现有模块未启用命名空间
   - 存在命名冲突风险

4. **缺乏业务状态管理**
   - 商品、订单、营销等核心业务缺乏专门的状态管理
   - 业务数据多通过组件内部状态管理

5. **缺乏状态持久化策略**
   - 仅在app模块中使用Cookie存储sidebar状态
   - 用户登录状态通过utils/auth.js管理，未与store集成

## 优化目标架构

### 目标Store结构
```
src/store/
├── index.js              # Store根配置文件
├── getters.js            # 根级别快捷getters
├── modules/
│   ├── core/             # 核心模块
│   │   ├── app.js        # 应用状态（优化版）
│   │   ├── user.js       # 用户认证（优化版）
│   │   ├── permission.js # 权限管理（优化版）
│   │   └── ui.js         # 界面状态（新增）
│   ├── business/         # 业务模块
│   │   ├── pms.js        # 商品管理模块
│   │   ├── oms.js        # 订单管理模块
│   │   ├── sms.js        # 营销管理模块
│   │   └── ums.js        # 用户权限管理模块
│   └── features/         # 功能模块
│       ├── common.js     # 通用状态
│       ├── cache.js      # 缓存管理
│       └── notification.js # 消息通知
├── plugins/              # Store插件
│   ├── persistence.js    # 持久化插件
│   └── logger.js         # 日志插件
└── utils/
    ├── storage.js        # 存储工具
    └── cache.js          # 缓存工具
```

## 迁移策略

### 阶段一：核心模块优化（向后兼容）

#### 1. 启用命名空间
- 为现有模块添加 `namespaced: true`
- 保持原有API兼容性

#### 2. 优化app模块
- 扩展状态：添加语言、主题、加载状态
- 完善持久化：使用localStorage替代Cookie
- 添加响应式设备检测

#### 3. 优化user模块
- 扩展用户信息字段
- 添加登录时间和活动时间跟踪
- 完善权限缓存机制

#### 4. 优化permission模块
- 完善路由权限管理
- 添加按钮级权限控制
- 优化菜单排序逻辑

#### 5. 新增ui模块
- 管理面包屑导航
- 标签页状态管理
- 布局状态控制

### 阶段二：业务模块重构

#### 1. 创建PMS商品管理模块
- 商品列表状态管理
- 分类树状态缓存
- 品牌和属性数据管理

#### 2. 创建OMS订单管理模块
- 订单列表和分页状态
- 订单统计数据
- 退货申请状态管理

#### 3. 创建SMS营销管理模块
- 优惠券活动状态
- 秒杀活动管理
- 广告投放状态

#### 4. 创建UMS用户权限管理模块
- 管理员列表状态
- 角色权限映射
- 资源权限管理

### 阶段三：功能模块完善

#### 1. 通用模块(common)
- 数据字典管理
- 系统配置缓存
- 常量数据管理

#### 2. 缓存模块(cache)
- 统一缓存策略
- 智能缓存失效
- 内存和本地存储管理

#### 3. 消息通知模块(notification)
- 系统消息管理
- 通知状态跟踪
- 未读消息计数

## 兼容性保证

### 1. 渐进式迁移
- 保持现有模块API不变
- 新功能通过新模块提供
- 逐步迁移组件使用新的状态管理

### 2. 别名支持
```javascript
// 在getters.js中保持向后兼容
const getters = {
  // 保留原有getters
  sidebar: state => state.app.sidebar,
  device: state => state.app.device,
  token: state => state.user.token,
  
  // 新增命名空间访问方式的别名
  'app/sidebar': state => state.app.sidebar,
  'app/device': state => state.app.device,
  'user/token': state => state.user.token,
}
```

### 3. 迁移工具
- 提供组件迁移脚本
- 状态访问方式转换工具
- 向后兼容性检查工具

## 回滚计划

如果迁移过程中出现严重问题：

1. **立即回滚步骤**
   - 还原store/index.js到原始版本
   - 移除新增模块文件
   - 恢复原始getters.js

2. **数据恢复**
   - 清除新增的localStorage数据
   - 恢复原有Cookie设置
   - 重置用户会话状态

3. **验证步骤**
   - 检查登录功能
   - 验证菜单权限
   - 确认核心业务功能

此迁移策略确保了平滑过渡，最小化风险，并为未来的功能扩展奠定了坚实基础。
